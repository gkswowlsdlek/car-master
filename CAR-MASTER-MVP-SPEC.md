# Car-Master MVP 기획·설계 문서

- 문서 상태: 1차 프로토타입 범위 승인 v0.2
- 작성일: 2026-07-13
- 목적: 요구사항 합의 후 단계별 구현을 시작하기 위한 기준 문서
- 현재 단계: **샘플 데이터 기반 클릭 가능한 프론트엔드 프로토타입 구현**

> 2026-07-13 승인 메모: 아래 전체 설계는 정식 MVP 확장 기준으로 보존한다. 1차 구현은 실제 Supabase·PG·지도 API·실시간 채팅을 제외하고, 랜딩과 역할별 핵심 거래 흐름을 샘플 데이터로 시연하는 프론트엔드 프로토타입으로 제한한다. 시공점 가입·패키지 참여는 관리자 승인, 딜러는 이메일 인증 후 즉시 이용, 순차 단일 요청, 48시간 응답기한, 전국 공통 단일 패키지 가격, 부가세 포함 원화 정수, 테스트 플랫폼 이용료 3%와 별도 PG 수수료 설정, 원 단위 반올림, 전액 환불만 지원하는 정책을 승인했다.

## 0. 먼저 확인할 모호점과 충돌 가능성

아래 항목은 구현 방식과 운영 정책에 직접 영향을 준다. 현재 문서에서는 `권장안`을 설계 기준으로 제안하지만, 승인 전까지 확정하지 않는다.

| Q22 | 채팅 범위 | 파일·이미지 첨부, 읽음 표시, 삭제, 알림 방식이 정해지지 않았다. | MVP는 실시간 텍스트 채팅, 시간 표시, 시스템 메시지만 제공한다. 이미지·파일·읽음 확인·푸시는 후속으로 미룬다. |
| Q23 | 계정 정지 효과 | 진행 중 거래가 있는 계정을 정지할 때 처리 정책이 없다. | 신규 로그인·요청·수락은 차단하되 기존 거래 자료는 보존하고 관리자가 거래를 마무리하도록 한다. |

## 1. MVP 범위 요약

### 1.1 제품 목표

Car-Master MVP는 리스·렌트 딜러가 고객 지역의 승인된 자동차 시공점을 검색하고, 관리자가 만든 표준 신차 패키지를 기준으로 시공을 요청한 뒤, 시공점과 일정을 조율하고 테스트 결제·진행·정산까지 한 플랫폼에서 검증하는 B2B 거래 시스템이다.

MVP가 검증해야 할 핵심 가설은 다음과 같다.

1. 딜러는 카카오톡 중개보다 지역·패키지 기반 시공점 검색과 요청을 선호한다.
2. 시공점은 표준 패키지 가격과 예상 일정 범위를 보고 요청을 수락할 수 있다.
3. 정확한 출고일이 미정인 거래도 `예상 범위 → 채팅 조율 → 일정 합의 → 결제` 흐름으로 관리할 수 있다.
4. 관리자는 거래·결제·수수료·정산 정보를 수기 장부보다 일관되게 관리할 수 있다.

### 1.2 포함 범위

#### 공통

- 이메일·비밀번호 기반 회원가입과 로그인
- 딜러·시공점·관리자 역할 분리와 접근 제어
- 한국어 입력 검증·오류 메시지
- 계정 상태 확인 및 정지 계정 차단

#### 딜러

- 프로필 및 소속 회사명 관리
- 차량 정보 등록·수정·조회
- 지역·지도 기반 승인 시공점 검색
- 시공점 상세와 참여 패키지 조회
- 예상 입고일 범위 기반 시공 요청
- 요청 수락 후 거래별 텍스트 채팅
- 일정 제안 확인·동의
- 테스트 결제 성공·실패 시나리오
- 예약·시공 진행 상태 및 거래내역 확인
- 결제 후 취소·환불 요청

#### 시공점

- 프로필, 사업자 정보, 주소, 좌표, 영업정보, 취급 필름·서비스 등록
- 사업자등록증 업로드
- 관리자 가입 승인 상태 확인
- 표준 패키지 참여 신청 및 승인 상태 확인
- 요청 수락·거절
- 거래별 텍스트 채팅과 일정 제안
- 입고 예정·입고 완료·시공 중·시공 완료 상태 변경
- 거래별 예상 정산금액과 정산내역 확인

#### 관리자

- 딜러·시공점 조회, 시공점 가입 승인·반려, 계정 정지·해제
- 표준 패키지와 구성항목 생성·수정·비활성화
- 시공점의 패키지 참여 승인·반려·중단
- 플랫폼 이용료율 및 테스트 PG 수수료 설정
- 거래·결제·취소·환불·정산 상태 조회 및 관리
- 거래 기반 신고 접수·처리

### 1.3 MVP 제외 범위

- 실제 PG 승인, 카드정보 저장, 실제 분리정산·자동 송금
- 세금계산서·현금영수증 발급
- 부분 결제·부분 환불·쿠폰·포인트
- 일반 소비자 회원과 소비자 직접 예약
- 시간 단위 예약 캘린더와 시공점 작업 베이 자동 배정
- 딜러 에이전시 조직·직원·다단계 권한 관리
- 여러 시공점 동시 견적·입찰
- 시공점별 패키지 가격 협상과 옵션 추가 결제

### 1.4 MVP 성공 지표 제안

- 딜러의 검색 시작 대비 시공 요청 전환율
- 시공점의 요청 응답률 및 평균 응답 시간
- 요청 대비 수락률
- 수락 대비 일정 합의율
- 일정 합의 대비 테스트 결제 성공률
- 결제 성공 대비 시공 완료율
- 관리자 1인당 거래 처리 소요시간 감소
- 취소·분쟁·신고율

## 2. 사용자별 상세 흐름

### 2.1 딜러 흐름

#### A. 가입과 초기 설정

1. 딜러 역할을 선택하고 이메일·비밀번호로 가입한다.
2. 이메일 인증 후 로그인한다.
3. 이름, 연락처, 소속 에이전시명 등 최소 프로필을 입력한다.
4. 딜러 대시보드에서 진행 중 거래와 최근 거래를 확인한다.

#### B. 시공점 탐색과 요청

1. 차량 제조사·모델·연식 등 차량 정보를 등록하거나 기존 차량을 선택한다.
2. 고객이 원하는 시공 지역을 주소 검색으로 입력한다.
3. 지도와 목록에서 승인된 시공점을 거리·지역 기준으로 확인한다.
4. 시공점 상세에서 주소, 영업정보, 취급 필름·서비스, 참여 중인 승인 패키지를 본다.
5. 원하는 패키지와 시공점을 선택한다.
6. 예상 입고 시작일과 종료일을 입력한다. 시작일은 종료일보다 늦을 수 없다.
7. 차량·지역·시공점·패키지 가격·예상 범위를 최종 확인하고 요청한다.
8. 요청은 `응답 대기`가 되고, 시공점이 거절하거나 기한 내 응답하지 않으면 다른 시공점에 재요청할 수 있다.

#### C. 일정 조율과 결제

1. 시공점이 요청을 수락하면 거래 전용 채팅방이 열린다.
2. 딜러와 시공점이 텍스트 채팅으로 세부 일정을 논의한다.
3. 시공점이 정확한 입고 예정일과 출고 예정일을 시스템상 제안한다.
4. 딜러가 제안을 동의하거나 재조정을 요청한다.
5. 동의 후 테스트 결제를 진행한다.
6. 결제 성공 시 예약이 확정되고, 실패 시 다시 결제할 수 있다.

#### D. 시공 진행과 종료

1. 딜러는 `예약 확정 → 입고 예정 → 입고 완료 → 시공 중 → 시공 완료` 진행 상태를 본다.
2. 시공점의 상태 변경은 채팅방에 시스템 메시지로 남는다.
3. 시공점이 시공 완료 처리하면 딜러가 완료를 확인한다.
4. 완료된 거래의 결제금액과 처리 이력을 거래내역에서 확인한다.
5. 문제가 있으면 거래 단위 신고 또는 환불 요청을 등록한다.

### 2.2 시공점 흐름

#### A. 가입과 심사

1. 시공점 역할을 선택해 회원가입하고 이메일을 인증한다.
2. 상호, 대표자명, 사업자등록번호, 연락처, 주소, 영업시간을 입력한다.
3. 주소는 카카오 주소 검색으로 좌표화한다.
4. 취급 필름과 제공 서비스를 선택·입력하고 사업자등록증을 업로드한다.
5. 가입 승인을 신청한다.
6. 심사 중에는 로그인과 정보 수정만 가능하다.
7. 관리자가 승인하면 검색 노출 및 패키지 참여 신청이 가능해진다. 반려 시 사유를 확인하고 수정 후 재신청한다.

#### B. 패키지 참여

1. 관리자가 활성화한 표준 패키지 목록을 조회한다.
2. 수행 가능한 패키지에 참여 신청한다.
3. 관리자가 승인한 패키지만 해당 시공점의 상세 화면과 검색 결과에 노출된다.
4. 참여 중단을 요청할 수 있으나 이미 진행 중인 거래에는 영향을 주지 않는다.

#### C. 요청 처리와 일정 조율

1. 새 시공 요청에서 차량, 패키지, 예상 입고 범위, 지역을 확인한다.
2. 응답 기한 내 수락 또는 거절하며, 거절 시 사유를 선택 또는 입력한다.
3. 수락하면 거래 채팅방이 열린다.
4. 채팅으로 조율한 정확한 입고 예정일과 출고 예정일을 입력해 딜러에게 제안한다.
5. 딜러의 동의와 결제 성공을 기다린다.
6. 결제 성공 전까지 예약은 확정되지 않는다.

#### D. 시공과 정산

1. 예약 확정 후 `입고 예정`으로 전환한다.
2. 차량 도착 시 `입고 완료`, 작업 시작 시 `시공 중`, 작업 종료 시 `시공 완료`로 변경한다.
3. 허용되지 않은 역방향 변경은 직접 수행할 수 없고 관리자 사유 입력이 필요하다.
4. 거래 상세에서 결제금액, 플랫폼 이용료, PG 수수료, 정산 예정금액을 확인한다.
5. 딜러의 완료 확인 후 정산 대기 상태가 된다.
6. 관리자가 외부 송금 후 지급완료로 표시한 정산내역을 확인한다.

### 2.3 관리자 흐름

#### A. 회원과 시공점 심사

1. 관리자 전용 계정으로 로그인한다.
2. 가입 심사 목록에서 시공점의 사업자·주소·서비스·첨부 서류를 확인한다.
3. 승인 또는 사유를 포함한 반려를 처리한다.
4. 사용자 상태와 거래 이력을 확인하고 필요한 경우 계정을 정지·해제한다.

#### B. 패키지 운영

1. 패키지명, 설명, 가격, 구성항목, 노출 상태를 등록한다.
2. 시공점의 패키지 참여 신청을 승인·반려한다.
3. 패키지를 수정하면 신규 요청부터 새 내용이 적용되고 기존 거래는 저장된 스냅샷을 유지한다.
4. 패키지 비활성화 시 신규 노출과 요청만 막고 기존 거래는 유지한다.

#### C. 거래·결제·환불 관리

1. 전체 거래를 상태, 기간, 딜러, 시공점 기준으로 조회한다.
2. 테스트 결제 성공·실패·취소·환불 이력을 확인한다.
3. 환불 요청의 사유와 현재 시공 상태를 확인하고 승인·거절한다.
4. 상태를 수동 보정할 때 반드시 사유와 변경 이력을 기록한다.

#### D. 수수료·정산 관리

1. 플랫폼 이용료율과 테스트 PG 수수료 기준을 변경한다.
2. 변경된 설정은 이후 결제 건부터 적용한다.
3. 시공 완료·딜러 확인이 끝난 거래의 정산 예정액을 검토한다.
4. 외부에서 송금한 뒤 정산을 지급완료로 표시하고 지급일·메모를 기록한다.

#### E. 신고 관리

1. 거래 기반 신고 내용을 조회한다.
2. 당사자와 거래·채팅 이력을 확인한다.
3. 처리 중, 해결, 기각 상태와 관리자 메모를 남긴다.
4. 필요하면 계정을 정지한다.

## 4. 데이터베이스 테이블과 관계

### 4.1 설계 원칙

- Supabase Auth 사용자의 식별자는 `auth.users.id`를 기준으로 한다.
- 역할과 업무 데이터는 공개 스키마의 테이블에 분리한다.
- 모든 업무 테이블에 필요한 경우 `created_at`, `updated_at`을 둔다.
- 삭제가 거래 기록을 훼손할 수 있는 항목은 물리 삭제 대신 상태·비활성화로 관리한다.
- 금액은 부동소수점이 아닌 원화 정수(`bigint`)로 저장한다.
- 주소 검색에 필요한 위도·경도는 별도 숫자 필드로 저장한다.
- 거래 생성 이후 바뀔 수 있는 패키지·가격·수수료 정보는 스냅샷으로 보존한다.
- 권한은 UI뿐 아니라 Supabase RLS 정책으로도 제한한다.

### 4.2 사용자·업체 영역

| 테이블 | 주요 필드 | 설명 |
|---|---|---|
| `profiles` | `id`, `role`, `display_name`, `phone`, `account_status` | Auth 사용자 공통 프로필. 역할은 dealer/shop/admin |
| `dealer_profiles` | `user_id`, `agency_name` | 딜러 전용 정보. MVP에서는 회사 조직 테이블을 두지 않음 |
| `shop_profiles` | `user_id`, `business_name`, `representative_name`, `business_number`, `phone`, `postal_code`, `address1`, `address2`, `latitude`, `longitude`, `business_hours`, `description`, `approval_status`, `rejection_reason`, `approved_at`, `approved_by` | 시공점과 심사 정보 |
| `shop_documents` | `id`, `shop_id`, `document_type`, `storage_path`, `review_status` | 사업자등록증 등 비공개 서류 메타데이터 |
| `service_catalog` | `id`, `name`, `category`, `is_active` | 필름 브랜드·서비스 기준 목록 |
| `shop_services` | `shop_id`, `service_id`, `custom_label` | 시공점 취급 품목 연결 |
| `vehicles` | `id`, `dealer_id`, `manufacturer`, `model`, `model_year`, `trim`, `color`, `vehicle_number`, `memo`, `is_active` | 딜러 소유 차량 정보. 고객 개인정보는 저장하지 않음 |

### 4.3 패키지 영역

| 테이블 | 주요 필드 | 설명 |
|---|---|---|
| `packages` | `id`, `name`, `description`, `price_krw`, `status`, `created_by` | 관리자가 운영하는 표준 패키지 |
| `package_items` | `id`, `package_id`, `name`, `description`, `sort_order` | 패키지 구성항목 |
| `shop_package_participations` | `id`, `shop_id`, `package_id`, `status`, `applied_at`, `reviewed_at`, `reviewed_by`, `rejection_reason` | 시공점의 참여 신청·승인 상태 |

관계:

- `packages 1:N package_items`
- `shop_profiles N:M packages`를 `shop_package_participations`가 연결한다.
- 딜러 검색에는 시공점 승인 상태와 패키지 활성 상태, 참여 승인 상태가 모두 유효한 조합만 노출한다.

### 4.4 요청·거래·채팅 영역

| 테이블 | 주요 필드 | 설명 |
|---|---|---|
| `service_requests` | `id`, `dealer_id`, `shop_id`, `vehicle_id`, `package_id`, `expected_arrival_start`, `expected_arrival_end`, `status`, `response_due_at`, `rejection_reason`, `package_snapshot`, `package_price_krw` | 딜러의 시공 요청과 당시 패키지 스냅샷 |

관계:

- `service_requests 0..1:1 transactions`: 수락된 요청만 거래를 만든다.
- `transactions 1:1 chat_rooms`, `chat_rooms 1:N chat_messages`
- `transactions 1:N schedule_proposals`
- `transactions 1:N transaction_status_history`

### 4.5 결제·정산 영역

| 테이블 | 주요 필드 | 설명 |

| `settlements` | `id`, `transaction_id`, `shop_id`, `gross_amount_krw`, `platform_fee_rate`, `platform_fee_krw`, `pg_fee_rate`, `pg_fixed_fee_krw`, `pg_fee_krw`, `refund_amount_krw`, `net_amount_krw`, `status`, `paid_at`, `admin_memo` | 거래별 정산 스냅샷 |

| `settlement_status_history` | `id`, `settlement_id`, `from_status`, `to_status`, `changed_by`, `reason`, `created_at` | 정산 상태 이력 |

정산 계산 제안:

`플랫폼 이용료 = 결제금액 × 거래에 저장된 플랫폼 이용료율`

`PG 수수료 = 결제금액 × 거래에 저장된 PG 수수료율 + PG 정액 수수료`

`시공점 정산 예정금액 = 결제금액 - 환불금액 - 플랫폼 이용료 - PG 수수료`

계산 결과가 음수가 되지 않도록 검증하며, 각 항목은 계산 당시 금액을 저장하여 정책 변경 후에도 기존 거래가 달라지지 않게 한다.

### 4.8 권한·RLS 기준

- 딜러는 자신의 프로필·차량·요청·거래·결제 결과만 조회·변경한다.
- 시공점은 자신의 업체 정보와 자신에게 온 요청·거래·정산만 조회·변경한다.
- 시공점은 승인 전 패키지 참여·요청 수락을 할 수 없다.
- 채팅 참가자는 해당 거래의 딜러, 시공점, 감사 목적의 관리자만 가능하다.
- 수수료 정책, 패키지 승인, 가입 승인, 환불 승인, 정산 지급완료는 관리자만 변경한다.
- 관리자 여부는 클라이언트가 보낸 역할값이 아니라 서버가 검증한 프로필과 정책으로 판단한다.
- 사업자등록증 저장소는 비공개 버킷으로 두고 서명된 임시 URL만 발급한다.

## 5. 예약 상태값 설계

예약 관련 상태를 하나의 거대한 값으로 두지 않고 `요청 상태`와 `거래·시공 상태`를 분리한다. 그래야 거절·만료와 실제 시공 진행을 명확하게 구분할 수 있다.

### 5.1 시공 요청 상태 `service_requests.status`

| 값 | 한국어 표시 | 의미 | 다음 허용 상태 |
|---|---|---|---|
| `pending` | 응답 대기 | 시공점 응답을 기다림 | `accepted`, `rejected`, `cancelled`, `expired` |
| `accepted` | 요청 수락 | 시공점이 수락하고 거래·채팅 생성 | 종료 상태 |
| `rejected` | 요청 거절 | 시공점이 거절 | 종료 상태 |
| `cancelled` | 요청 취소 | 결제 전 딜러 또는 허용된 주체가 취소 | 종료 상태 |
| `expired` | 응답 만료 | 응답 기한 초과 | 종료 상태 |

### 5.2 거래·예약·시공 상태 `transactions.status`

| 값 | 한국어 표시 | 상태 변경 주체 | 전이 조건 |
|---|---|---|---|
| `schedule_coordinating` | 일정 조율 중 | 시스템 | 요청 수락 시 생성 |
| `schedule_agreed` | 일정 합의 | 딜러 | 최신 일정 제안 동의 시 |
| `payment_pending` | 결제 대기 | 시스템 | 일정 합의 직후 |
| `reservation_confirmed` | 예약 확정 | 시스템 | 테스트 결제 성공 시 |
| `arrival_expected` | 입고 예정 | 시공점 | 예약 확정 후 |
| `vehicle_received` | 입고 완료 | 시공점 | 차량 실제 입고 시 |
| `work_in_progress` | 시공 중 | 시공점 | 작업 시작 시 |
| `work_completed` | 시공 완료 | 시공점 | 작업 종료 시 |
| `completion_confirmed` | 완료 확인 | 딜러/관리자 | 딜러 확인 또는 관리자 처리 시 |
| `cancelled` | 거래 취소 | 시스템/관리자 | 결제 전 취소 또는 결제 취소 완료 시 |
| `refund_pending` | 환불 처리 중 | 시스템 | 결제 후 환불 요청 승인 절차 시작 |
| `refunded` | 환불 완료 | 시스템/관리자 | 전액 환불 성공 시 |
| `disputed` | 분쟁 처리 중 | 관리자 | 신고·분쟁으로 진행 일시 중지 |

### 5.3 상태 전이 원칙

정상 흐름:

`일정 조율 중 → 일정 합의 → 결제 대기 → 예약 확정 → 입고 예정 → 입고 완료 → 시공 중 → 시공 완료 → 완료 확인`

- 시공점은 정상 순서의 다음 단계로만 변경한다.
- 입고 완료 이후의 취소·환불은 자동 처리하지 않고 관리자 검토로 보낸다.
- 관리자의 예외 변경은 가능하되 사유와 전후 상태를 감사 로그에 남긴다.
- 채팅 메시지는 거래 상태와 별개로 보존한다.
- `분쟁 처리 중`에는 시공 상태와 정산을 잠그고 관리자만 해제한다.

### 5.4 가입·패키지 참여 보조 상태

- 시공점 승인: `draft`, `pending`, `approved`, `rejected`, `suspended`
- 패키지: `draft`, `active`, `inactive`
- 시공점 패키지 참여: `pending`, `approved`, `rejected`, `withdrawn`, `suspended`

## 6. 결제 및 정산 상태값 설계

결제 주문, 개별 결제 시도, 환불, 정산은 서로 다른 생명주기를 가지므로 별도 상태로 관리한다.

### 6.1 결제 주문 `payment_orders.status`

| 값 | 한국어 표시 | 의미 |
|---|---|---|
| `pending` | 결제 대기 | 아직 성공한 결제 없음 |
| `paid` | 결제 완료 | 한 번의 결제 시도가 성공 |
| `cancel_pending` | 취소 처리 중 | 결제 취소 요청 처리 중 |
| `cancelled` | 결제 취소 | 시공 전 승인 취소 완료 |
| `refund_pending` | 환불 처리 중 | 환불 승인·처리 중 |
| `refunded` | 환불 완료 | 전액 환불 완료 |

결제 실패는 주문 전체를 종료시키지 않는다. 주문은 `pending`을 유지하고 각 실패 기록은 `payment_attempts`에 남겨 재시도를 허용한다.

### 6.2 결제 시도 `payment_attempts.status`

| 값 | 한국어 표시 | 의미 |
|---|---|---|
| `ready` | 결제 준비 | 테스트 결제 요청 생성 |
| `processing` | 결제 처리 중 | 중복 요청을 막는 처리 구간 |
| `succeeded` | 결제 성공 | 테스트 승인 성공 |
| `failed` | 결제 실패 | 테스트 승인 실패, 실패 코드 저장 |
| `cancelled` | 승인 취소 | 성공 결제의 취소 완료 |

테스트 결제 인터페이스는 추후 실제 PG 구현체로 교체할 수 있게 `결제 요청`, `결제 조회`, `승인 취소`, `환불` 책임을 분리한다. DB에는 카드번호, CVC, 비밀번호, 전체 유효기간 등 민감정보를 저장하지 않는다.

### 6.3 환불 `refunds.status`

| 값 | 한국어 표시 | 의미 |
|---|---|---|
| `requested` | 환불 요청 | 딜러가 요청함 |
| `reviewing` | 검토 중 | 관리자가 처리 중 |
| `approved` | 환불 승인 | 테스트 환불 실행 가능 |
| `rejected` | 환불 거절 | 사유 기록 후 종료 |
| `processing` | 환불 처리 중 | 테스트 모듈 처리 중 |
| `succeeded` | 환불 완료 | 전액 환불 완료 |
| `failed` | 환불 실패 | 재처리 또는 관리자 확인 필요 |

### 6.4 정산 `settlements.status`

| 값 | 한국어 표시 | 진입 조건 |
|---|---|---|
| `estimated` | 정산 예정 | 결제 성공 시 수수료 스냅샷으로 계산 |
| `on_hold` | 정산 보류 | 환불·분쟁·관리자 보류 |
| `ready` | 정산 대기 | 완료 확인 및 환불·분쟁 없음 |
| `processing` | 정산 처리 중 | 관리자가 외부 송금 처리 시작 |
| `paid` | 지급 완료 | 외부 송금 후 관리자 확인 |
| `cancelled` | 정산 취소 | 전액 취소·환불로 지급 대상 없음 |
| `failed` | 정산 실패 | 외부 송금 실패 기록 |

### 6.5 금액·멱등성 규칙

- 동일 거래의 성공 결제는 1건만 허용한다.
- 결제·취소·환불 명령에는 고유 요청 키를 사용해 중복 처리를 막는다.
- 수수료율은 퍼센트 문자열이 아닌 충분한 정밀도의 숫자로 저장하고 금액 계산 결과는 원 단위 정수로 확정한다.
- 환불 중이거나 분쟁 중이면 정산을 자동으로 `on_hold` 처리한다.
- 지급완료 정산을 되돌릴 때는 기존 레코드를 덮어쓰기보다 조정 이력을 추가하는 방향으로 확장 가능하게 설계한다.

5. **5차 확인:** 채팅·일정 합의·테스트 결제
6. **6차 확인:** 시공 상태·정산·관리자 운영
7. **최종 확인:** 테스트, 샘플 데이터, 배포 문서

### 7.2 기술상 주요 위험

- 역할별 RLS가 잘못되면 다른 업체의 거래·채팅·정산이 노출될 수 있다.
- 카카오맵의 브라우저 키와 서버 REST 키 사용 범위, 허용 도메인 설정을 구분해야 한다.
- 실시간 채팅은 권한 정책과 재접속·중복 메시지 처리 검증이 필요하다.
- 결제·환불·정산 상태가 한 필드에 섞이면 회계 추적이 어려워 반드시 분리해야 한다.
- 패키지 가격과 수수료율을 현재 설정에서 즉시 계산하면 과거 거래 금액이 바뀌므로 스냅샷이 필요하다.
- 관리자 수동 보정은 편리하지만 감사 로그와 사유 입력이 없으면 분쟁 대응이 어렵다.

## 8. 추가 결정이 필요한 질문

### 8.1 1차 승인 전에 반드시 결정

1. 시공점의 패키지 참여 신청은 관리자가 건별 승인하는 것이 맞는가?
2. MVP에서는 한 차량·패키지당 한 시공점에만 순차 요청하는 방식으로 확정해도 되는가?
3. 예약 확정은 `시공점 일정 제안 → 딜러 동의 → 결제 성공` 시점으로 보아도 되는가?
4. 요청 응답 기한 기본값을 48시간으로 둘 것인가?
5. 패키지는 전국 공통 단일 가격인가, 시공점별 가격이 필요한가?
6. 표시 가격은 부가세 포함인가? 플랫폼 이용료의 부가세 처리는 어떻게 할 것인가?
7. PG 수수료는 시공점 정산금액에서 차감하는가? 테스트 PG 수수료 기본값은 얼마인가?
8. 수수료 원 단위 처리는 반올림, 올림, 버림 중 무엇을 적용할 것인가?
9. MVP 환불은 전액 환불만 지원해도 되는가?
10. 시공 완료 후 딜러의 완료 확인 기한과 미응답 시 자동 확정 여부는 어떻게 할 것인가?
11. 실제 정산 송금은 플랫폼 밖에서 처리하고 관리자가 지급완료만 기록하는 것이 맞는가?
12. 차량번호는 필수, 선택, 미수집 중 어느 정책으로 할 것인가?

### 8.2 화면·운영 상세 결정

13. 딜러 가입도 관리자 승인이 필요한가, 이메일 인증만으로 즉시 이용 가능한가?
14. 시공점 가입에 필요한 정확한 서류는 사업자등록증 1종이면 충분한가?
15. 시공점 검색 기본 반경과 정렬 기준은 무엇인가? 예: 10km·거리순.
16. 영업시간은 단순 텍스트인가, 요일별 구조화 입력인가?
17. 취급 필름·서비스는 관리자가 만든 목록만 선택하는가, 시공점 자유 입력도 허용하는가?
18. 일정에서 시간까지 입력할 것인가, 정확한 날짜만 합의할 것인가?
19. 요청 거절 사유를 딜러에게 그대로 공개할 것인가?
20. 채팅 알림은 MVP에서 웹 내부 알림만으로 충분한가?
21. 신고 카테고리와 계정 정지 기간은 어떤 운영 기준으로 정할 것인가?
22. 관리자 계정 최초 생성은 Supabase에서 수동 지정하는 방식으로 할 것인가?
23. 예비창업패키지 시연용으로 사용할 샘플 지역, 딜러 수, 시공점 수, 패키지 수는 어느 정도가 좋은가?

### 8.3 권장 기본 승인안

빠른 MVP 제작을 위해 다음 조합을 권장한다.

- 시공점 가입과 패키지 참여 모두 관리자 승인
- 딜러는 이메일 인증 후 즉시 이용
- 단일 시공점 순차 요청, 응답 기한 48시간
- 전국 공통 패키지 가격, 부가세 포함 원화 정수
- 일정 합의 후 결제 성공 시 예약 확정
- 플랫폼 수수료 기본 3%, 정책 테이블에서 변경
- 테스트 PG 수수료는 별도 관리자 설정, 시공점 정산에서 차감
- MVP 전액 환불만 제공, 결제 후 환불은 관리자 승인
- 실제 송금 없이 정산 상태와 외부 지급 결과만 기록
- 고객 이름·전화번호 미수집, 차량번호 선택 입력
- 텍스트 채팅과 웹 내부 상태 알림만 제공
- 시공점 10곳, 딜러 5명, 패키지 3개, 거래 시나리오 10건의 샘플 데이터

## 승인 기준

이 문서는 아래 세 가지 중 하나의 방식으로 승인·수정할 수 있다.

1. **권장안 전체 승인:** `권장 기본 승인안으로 진행`이라고 회신
2. **일부 수정 승인:** 질문 번호와 원하는 답을 회신. 예: `Q4는 24시간, Q12는 PG 3.3%`
3. **문서 보완 요청:** 추가하고 싶은 기능·정책·화면을 회신

승인 전에는 프로젝트 초기화나 기능 코드를 작성하지 않는다.
